# ADR-005: Application tracker persistence — SQLite via SQLAlchemy 2.0 + Alembic, documents as hashed files

## Status

Accepted — 2026-06-21

## Context

The tracker is the third layer of Dossier: a record of every job application, its status over
time, and — a primary requirement — the **exact CV and cover letter sent** with each application,
preserved for later analysis. It is being built ahead of the engine (M2–M4) that will eventually
*generate* those documents, so the tracker must also accept **manually-provided** documents now.

Constraints already fixed: the data is personal and lives in the private `dossier-data` repo,
located at runtime via `DOSSIER_DATA_PATH` (ADR-001); the README tentatively names **SQLite** as
the application-log store; the stack is Python 3.13 managed with `uv` (ADR-003), already using
Pydantic for the inventory. The project values modern, transferable practice and cautious
dependency choices.

Three coupled decisions are needed: how Python talks to SQLite; how the sent documents are
stored; and whether schema migrations exist from day one.

## Considered alternatives

### Persistence / access layer

**A. SQLAlchemy 2.0 ORM + Alembic (chosen).** The de-facto standard ORM with typed `Mapped[]`
models and first-class, autogenerated migrations (Alembic). Most transferable and portfolio-
valuable; robust schema evolution for precious personal data. Cost: more boilerplate than a
micro-approach, and two new dependencies. *Chosen.*

**B. SQLModel + Alembic.** SQLModel unifies SQLAlchemy and Pydantic, reducing boilerplate and
cohering with the inventory's Pydantic models. But it is a thinner, less-actively-maintained
layer that occasionally leaks to raw SQLAlchemy anyway. *Rejected* — prefer the mature substrate
directly for a learning project.

**C. Raw `sqlite3` (stdlib) + hand-rolled migrations.** Zero new dependencies; maximal control
and SQL learning; migrations via `PRAGMA user_version`. But every query and migration is written
and maintained by hand, which is more code and more room for error as the schema grows.
*Rejected* — the safety and standardisation of an ORM + Alembic outweigh the dependency cost.

### Storing the sent documents

**A. Files in `dossier-data` + DB stores path, sha256, and provenance (chosen).** Rendered docs
are copied to `applications/<id>/` and the DB row records a relative path, a content hash, and
provenance (model, language, inventory commit, timestamp). Human-readable, git-diffable, DB stays
small, and the hash makes the sent version tamper-evident. *Chosen.*

**B. Full content as TEXT in the DB.** Self-contained and queryable, but the DB becomes the only
copy and changes are not diffable in git. *Rejected.*

**C. Both file and content-in-DB.** Maximum robustness, at the cost of storing each document
twice. *Rejected as unnecessary* given `dossier-data` is already version-controlled.

### Migrations timing

**A. From day one (chosen).** Alembic is configured now, with the initial schema as the first
revision. The application log is irreplaceable personal data; safe, repeatable schema evolution
matters and demonstrates real practice. *Chosen.*

**B. Defer (create_all now, migrations later).** Less setup now, but the first real schema change
becomes a manual, risky migration of live data. *Rejected.*

## Decision

Persist the tracker in **SQLite** via **SQLAlchemy 2.0 ORM** with **Alembic** migrations from the
start. Model three tables — `applications`, `application_events` (status history), and
`documents`. Store each **sent document as an immutable file** under
`dossier-data/applications/<id>/`, with the DB recording its relative path, **sha256**, and
provenance (`model`, `language`, `inventory_commit`, `generated_at`).

Specifics:

- Models in `src/dossier/tracker/models.py`; status/kind are non-native SQL enums persisting the
  lowercase member *value* (e.g. `"applied"`, `"cv"`).
- Service functions in `src/dossier/tracker/repository.py` enforce the invariants: every status
  change appends an event; follow-up dates derive from `DOSSIER_FOLLOWUP_DAYS` and clear on
  terminal statuses; `attach_document` copies + hashes the file and best-effort captures the
  `dossier-data` git commit.
- A shared engine/session in `src/dossier/db.py` (with `PRAGMA foreign_keys=ON`).
- Alembic under `migrations/`; `env.py` resolves the URL from `dossier.config.get_database_url`
  (override via `DOSSIER_DATABASE_URL` / `-x db_url=`) — no hard-coded paths (ADR-001).
- A `dossier track` / `dossier db upgrade` CLI surface (argparse).
- New runtime dependencies: `sqlalchemy` (>=2.0), `alembic`.
- The SQLite file is committed in `dossier-data` (its `.gitignore` commits content).

## Consequences

### Positive

- Strong, standard data layer with safe, versioned schema evolution.
- The exact sent documents are preserved immutably and verifiably (hash), with enough provenance
  (model, language, inventory commit) to analyse *how* each was produced.
- A clean `dossier track` workflow that works today with manual documents and will accept
  engine-generated output unchanged later.

### Negative

- Two new runtime dependencies and ORM/migration boilerplate.
- The SQLite database is a binary blob in git; acceptable for a single-user, low-frequency log.

### Neutral / future options

- Status transitions are unconstrained for now; a strict state machine could be added later.
- If binary DB diffs become noisy, a pre-commit `sqlite3 .dump` text export can be introduced
  without changing the schema.
